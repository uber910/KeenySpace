# Authentication

KeenySpace authentication follows a similar model to Anthropic's API: OIDC for interactive
human users, and long-lived API keys for programmatic and agent access.

Authorization (roles, ACL, group mappings beyond the entry gate) is deferred to v1.5. v1
is single-org: all authenticated users see all workspaces.

## Authentication surfaces

### OIDC (interactive users)

Login flow:
1. User runs `keenyspace login` — opens a browser tab via the device-code flow.
2. Browser is directed to the OIDC IdP (Authentik in the reference setup).
3. IdP authenticates the user and returns an authorization code.
4. Server exchanges the code for tokens, validates the JWT, creates a session.
5. CLI receives a session token and stores it in `~/.config/keenyspace/auth.json` (mode 0600).

OIDC discovery is available at `GET /v1/api/auth/discovery`. Any standards-compliant OIDC
provider works at the protocol level; Authentik is the reference for v1.

### API keys (agents and programmatic access)

Format: `ks_live_<random-hex>`. Generated by the server, shown plaintext once on creation.

Usage: `Authorization: Bearer ks_live_...` in HTTP and MCP requests.

API keys are the **primary auth path for MCP agents**. OIDC bearer tokens are short-lived
(hours), which is incompatible with long-running MCP sessions. API keys do not expire by
default and are minted by admins.

API keys are stored as argon2id hashes. Lookup uses a sha256+pepper hash for fast index
lookup before bcrypt/argon2 verification.

### MCP authentication

MCP requests go through the same `CompositeAuthBackend` middleware as HTTP requests. The
single `Starlette AuthMiddleware` at the FastAPI root covers both transports. FastMCP tools
read the caller identity via `auth_bridge` from `get_http_request().state.user`. FastMCP's
built-in OAuthProxy is not used (incompatible with the mount-path architecture).

## Group entry gate (D-05)

Optionally, membership in a specific IdP group can be required for access. When
`KEENYSPACE_AUTH__REQUIRED_GROUP` is set, `CompositeAuthBackend.authenticate()` checks the
`groups` claim in the OIDC token after validation. Requests from users not in the required
group receive HTTP 401.

API key holders bypass the group gate — API keys are minted by admins and represent
already-authorized machine access.

## Trust model

- **Server admin is trusted.** SSH access to the server filesystem bypasses all auth. This
  is standard practice for self-hosted software (Postgres, Outline, GitLab).
- **Server trusts JWTs signed by the configured IdP.** JWKS are fetched from the IdP and
  cached with periodic refresh.
- **TLS is terminated externally** by the reverse proxy (Caddy or nginx). The server
  speaks plain HTTP internally.

## Auth middleware

Single `Starlette AuthMiddleware` on the FastAPI root app. It is installed before the MCP
mount (`app.mount("/v1/mcp", ...)`) so that all requests to both `/v1/api/*` and `/v1/mcp`
are covered by the same check. The middleware runs before route handlers and populates
`request.state.user` with the resolved `User` or raises `AuthenticationError`.

`/healthz` and `/readyz` are exempted from auth requirements.
